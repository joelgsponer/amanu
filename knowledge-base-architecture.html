What it is

Two memories, not one

The idea that makes it last is splitting memory in two — one for what your documents say, one for how you like things done.

Memory · the matter

The knowledge base

A wiki-style store over your documents: immutable sources plus a synthesised, cross-linked layer the agent keeps current. The single source of truth for what you know.

Memory · the method

Agent memory

The agent's own durable preferences and lessons — one fact per file, recalled when relevant. It shapes how every task is done, and compounds the longer you use it.

It's deliberately low-tech: plain Markdown, folders, a little YAML, and wiki-links. No database, no server, no lock-in — human-readable and yours, on your machine.

How it's structured

Three layers, plain files

Sources never change. The synthesis is freely rewritten. A single schema file is the constitution that ties them together.

your-knowledge-base/ ├─ SCHEMA.md the constitution — conventions & workflows ├─ inbox/ drop zone for new documents ├─ raw/ ▸ Layer 1 — immutable source files, by category │ └─ <your categories>/ └─ kb/ ▸ Layer 2 — the synthesis the agent owns ├─ entities/ people, orgs, accounts — one page each ├─ topics/ cross-cutting themes ├─ sources/ one summary page per document ├─ queries/ saved answers & reports └─ index.md · log.md · overview.md

Every page carries YAML frontmatter (so the store is queryable) and links related pages with [[wiki-links]] (so it's a navigable graph). Layer 3 is SCHEMA.md — read it and you can run the whole system.

What the agent does

Your tireless librarian

Three everyday workflows do the work — each a one-word command once the system is built.

ingest

File & cross-reference

Read a new document, classify it, move it into raw/, write a source page, and roll the facts up into the entities and topics it touches.

query

Answer from the synthesis

Answer a question from the kb/ pages — citing them — instead of re-reading raw files. Save anything worth keeping.

lint

Keep it honest

Sweep for contradictions, stale claims, orphan pages, and gaps, and report what needs a human's eye.

And it documents itself as it goes: a CHANGELOG.log records every decision and why; a kb/log.md tracks routine operations; and it writes CLAUDE.md + AGENTS.md so any future agent that opens the folder is instantly oriented.

The memory system

Method meets matter

The two memories cooperate. A lesson captured in agent memory shapes how the knowledge base is written; the knowledge base holds the facts that lessons are about.

method

Agent memory

One fact per file + an index. Recalled when a task makes it relevant — "ask before inferring", "always flag X", "summaries should be terse".

matter

Knowledge base

Sources → synthesis. The durable record of what your documents actually say, kept cross-referenced and current.

Some facts live in both — an anchor value recorded in the knowledge base and held in agent memory so any source that disagrees gets flagged. Together they turn a one-off summariser into a system that gets more useful over time.

How you set it up

Build it with an agent

You don't configure this by hand. You hand the agent one file and it builds the system with you — a guided, eleven-phase co-build that is the design, the onboarding, and the tutorial all at once.

P0 orientP1 interviewP2 scaffoldP3 schemaP4 seed memoryP5 first ingestP6 first queryP7 skillsP8 extensionsP9 lint + docsP10 handoff

Every phase runs the same honest rhythm — and stops leave the system working:

teach→tailor→build→verify→checkpoint→log

It's resumable (a machine-readable amanu.yaml manifest tracks where you are), grows in tiers (Starter → Connected → Automated → Intelligent), and handles secrets safely (.gitignore first, then a .env.example — values never committed).

The extension catalogue

28 ways to grow it

The core just files and answers. Extensions attach at the edges to feed it, surface it, automate it, and sharpen it. Each one below is a ready-to-use build prompt — copy it, hand it to your agent, done.

How to add any extension

  1. Open your built knowledge base with your agent. It auto-loads AGENTS.md / SCHEMA.md, so it already knows your conventions.
  2. Copy the extension's build prompt (the button on each card).
  3. Paste it and say "add this extension." The agent scaffolds the skill or hook wired to your system, sets up any secrets in .env, and updates the manifest + entrypoints.

Inbound — feed the store

Turn the world's documents into sources, automatically.

▸emailA Gmail label becomes an automatic source feed.inboundConnected

Needs: gog (Google Workspace CLI).  Secrets: none — gog uses its own OAuth login.

Create an `ingest-email` skill that lists Gmail messages labelled `ingest-pending`; for each (oldest first) writes the body to `kb/sources/` and downloads attachments to `inbox/`, runs `/ingest`, then relabels `ingest-pending`→`ingest-done`. Pull-only, idempotent (skip already-done), leave the label on failure so the next run retries.

Trust — never sends, replies, or forwards.

▸drive-syncA cloud folder mirrors into the inbox.inboundConnected

Needs: gog (Drive) or rclone.  Secrets: none for gog; rclone keeps its own config (out of git).

Create a `sync-drive` skill/hook that pulls new/changed files from a named remote folder into `inbox/`, records synced file IDs for idempotency, then runs `/ingest`.

Trust — read-only on the remote; never uploads.

▸scan-ocrScans and photos become searchable text.inboundConnected

Needs: ocrmypdf / tesseract.  Secrets: none.

Create a `scan-ocr` step that, for image/PDF files in `inbox/`, runs OCR to produce a text layer before `/ingest` reads them; keep the original in `raw/`.

Trust — fully local.

▸web-clipperA URL becomes a clean Markdown source page.inboundConnected

Needs: a fetch tool + pandoc/readability.  Secrets: none.

Create a `clip` skill that takes a URL, fetches and converts the main content to Markdown, writes it as a `kb/sources/` page with `source_url` in frontmatter, and links relevant entities/topics.

Trust — fetches public pages only; no private data leaves.

▸audio-transcribeVoice memos & meetings become text sources.inboundConnected

Needs: a transcription tool (e.g. whisper).  Secrets: none if local; an API key if hosted.

Create a `transcribe` step that turns audio files in `inbox/` into text, files the transcript as a `kb/sources/` page, and keeps the audio in `raw/`.

Trust — prefer a local model to keep audio on-machine.

▸calendar-pullCalendar events become dated context.inboundConnected

Needs: gog (Calendar).  Secrets: none.

Create a `pull-calendar` skill that imports events in a date window as dated notes/source pages, linked to the entities they involve.

Trust — read-only.

▸csv-importTabular & financial exports become structured sources.inboundConnected

Needs: a CSV parser (built-in).  Secrets: none.

Create an `import-csv` skill that parses a CSV/XLSX in `inbox/` into a normalized table page under `kb/sources/`, with column mapping confirmed by the user; never infers transaction meaning — record observations, ask before interpreting.

Trust — local; flag any secrets/account numbers for the user.

▸messaging-exportExported chat threads become sources.inboundConnected

Privacy & principles

Yours, and trustworthy

  • Two memories, kept apart. Method in agent memory, matter in the knowledge base.
  • Sources are immutable. The synthesis is freely rewritten; the originals never change.
  • Everything is plain files. Markdown, frontmatter, folders, version control — no database, nothing you can't open by hand.
  • Observe before interpreting. Ambiguous data is recorded as observed, and confirmed before it becomes a "fact".
  • Local-first, explicit egress. Nothing leaves your machine without a deliberate yes; inbound and outbound are never fused.
  • Generic until you personalise it. This blueprint ships anonymized — your categories, entities, and examples come from you, at build time.
