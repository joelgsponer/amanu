The Two-Memory Architecture

01The idea

Point a capable AI agent at a folder of your documents and give it a constitution. It becomes a tireless librarian: reading, summarising, cross-referencing, and keeping a synthesis current as new material arrives.

The trick that makes it durable is using two memories, not one. A knowledge base remembers what your documents say. A separate agent memory remembers how you want the work done — your preferences, your corrections, your standing instructions. Together they turn a one-off summarising tool into a system that compounds in value the longer you use it.

The pattern is deliberately low-tech: plain Markdown files, a folder hierarchy, YAML frontmatter, and wiki-style links. No database, no server, no lock-in. Everything is human-readable, version-controllable, and stays on your machine.

02Two cooperating memories

Keep them strictly separate. One stores method; the other stores matter.

Memory A · the method

Agent memory

The assistant's own persistent, cross-session memory: durable preferences, lessons learned, and standing instructions. It shapes how every task is performed. Small, append-mostly, auto-recalled when relevant.

Memory B · the matter

The knowledge base

A wiki-style store over your documents: immutable sources, a synthesised Markdown layer the agent owns, and a schema that governs both. Versioned, queryable, the single source of truth for what you know.

raw sourcessynthesisschema[[links]]

The two reinforce each other. A lesson captured in agent memory ("don't guess at ambiguous data — ask first") governs how knowledge-base pages get written. A stable fact may even live in both — recorded as data in the knowledge base, and held as an anchor in agent memory so any conflicting source gets flagged.

03Agent memory, in detail

Structurally trivial: a folder where each file holds exactly one fact, plus a single index listing them all.

The index is loaded into the agent's context at the start of every session. The individual fact files are pulled in only when they're relevant to the task at hand — so the memory can grow large without flooding context.

# the agent's per-project memory folder memory/ ├─ INDEX.md › one line per fact — loaded every session ├─ pref-concise-summaries.md › type: user ├─ ask-before-inferring.md › type: feedback └─ current-migration.md › type: project

Four kinds of fact

Every memory declares a type. The type decides what belongs in the file and how its body is shaped.

user

Stable facts about who you are — role, context, lasting preferences.

"Prefers concise, skimmable summaries."
feedback

How the agent should work — corrections and confirmed approaches. Body carries explicit Why and How to apply lines.

"Don't infer meaning from ambiguous data — ask first."
project

Ongoing work, goals or constraints not derivable from the files. Relative dates made absolute.

"Current focus through Q3: consolidate older records."
reference

Pointers to external resources — links, dashboards, identifiers worth keeping.

"Status board: <internal-link>."

Facts cross-link with [[name]] — the same convention the knowledge base uses — so the memory folder is itself a small graph. A link to a fact that doesn't exist yet is fine: it marks something worth writing later.

Anatomy of a fact

---
name: ask-before-inferring
description: Don't infer meaning from ambiguous data alone — observe, then ask.
type: feedback
---
When data is ambiguous, describe only what is observable; do not
assign meaning without confirmation.

Why:      Pattern-matching to a plausible story produces confident-but-wrong
          conclusions that then propagate across many pages.
How to apply: State the observation, ask one clarifying question, and
          wait before updating any downstream page.

Related: [[ingest-workflow]]

The description line does real work: it is the field matched against the current task to decide whether the fact is worth recalling. Vague descriptions never surface — specific ones earn their keep.

04How agent memory "compiles"

Compilation here means assembly: how scattered one-fact files become live context at the right moment — and how a session gets distilled back into a fact.

Two passes, with the index as the hinge between them. The read pass assembles memory into a session; the write pass condenses the session back into memory.

fig 1 — the index is the hinge: every read starts from it, every write ends by updating it

The index loads, always

The index enters context at session start — one line per fact, never the bodies. It is the table of contents the rest of the pass reads against.

Relevance is judged on the description

As work proceeds, the task is matched against each fact's description. Specific descriptions get recalled; vague ones never do.

Relevant facts are injected — as context, not commands

A matched fact is surfaced as background information, carrying a reminder that it is a point-in-time note to verify, not live ground truth.

Writing back is the reverse

A durable lesson is condensed to one fact, deduplicated against existing files (update beats duplicate), written with full frontmatter, and registered with a one-line pointer in the index. Facts that prove wrong are deleted.

Treat recalled facts as leads, not law. Memories are point-in-time observations. A fact may name a file or figure that has since changed — so on recall it is verified against current sources before being acted on.

05The knowledge base

A three-layer store that turns a pile of documents into a cross-referenced, queryable synthesis.

Layer 1 · sources

  • The original documents — files, scans, exports
  • Organised into categories you define for your domain
  • Immutable — never edited, only renamed or moved
  • Treated as authoritative ground truth

Layer 2 · synthesis

  • The Markdown layer the agent owns entirely
  • One summary page per source; rolled-up entity & topic pages
  • An index catalog, an op log, a dashboard, saved analyses
  • Always kept current as new sources arrive

A third layer — call it the schema — is a single instruction file that defines the directory layout, naming conventions, the frontmatter spec, and the workflows. It is the agent's constitution, and it co-evolves with the store rather than sitting frozen above it.

knowledge-base/ ├─ SCHEMA.md › the agent's constitution — conventions & workflows ├─ inbox/ › drop zone for new sources awaiting ingest ├─ raw/ › immutable source documents │ └─ <category>/ › your domain's categories (define your own) └─ kb/ › the synthesised knowledge layer ├─ index.md › catalog of every page, by category ├─ log.md › append-only chronological op log ├─ overview.md › "current state of affairs" dashboard ├─ entities/ › people, orgs, accounts — 1 page each ├─ topics/ › cross-cutting themes ├─ sources/ › 1 summary page per ingested document └─ queries/ › saved analyses, comparisons, reports

The connective tissue is the same idea as agent memory, scaled up: YAML frontmatter on every page (so the store is queryable by category, date, or tag) and [[wiki-links]] through the prose (so entities, topics and sources form one navigable graph). When the agent mentions something that lacks a page, it creates a stub rather than leaving a dead reference.

06The ingest pipeline

Where agent memory compiles a lesson into a fact, the knowledge base compiles a raw document into synthesised, cross-linked knowledge. The recipe is deterministic.

Read & classify

A file lands in inbox/. The agent reads it (text and PDFs natively; images via OCR) and classifies category, type, date, and the parties involved.

Surface anything notable first

Anything surprising, anomalous, or contradicting an existing page is raised before filing — never summarised and stored silently.

Rename & move into raw/

Into raw/<category>/ under a consistent, sortable naming convention; near-duplicates are detected and reconciled.

Write the source page

A kb/sources/ page mirroring the filename — summary, key facts, notes, cross-references — built from a fixed template.

Roll up into entities & topics

The affected entity and topic pages are updated: a line added, the source linked, totals and summaries revised where the document changes them. This is the synthesis step.

Record & index

A line appended to the op log; the new source added to the catalog. The index stays current; nothing becomes an orphan.

Two companion workflows keep the synthesis honest. Query reads the catalog first, then the most relevant pages (preferring synthesis over re-reading raw files), and offers to save substantial answers into queries/. Lint sweeps for contradictions, stale claims, orphan pages, implicit entities that lack a page, missing cross-references, and data gaps — saving a dated report.

07How to build it yourself

You need an agentic assistant with three capabilities: it can read and write files, it can read documents (PDFs, images), and it has a persistent cross-session memory. Given that, the rest is convention.

Create the folders

Make inbox/, raw/, and kb/ (with entities/ topics/ sources/ queries/). Put it under version control from day one — every operation becomes a reviewable commit.

Write the schema — the constitution

This single instruction file is what turns a generic assistant into a consistent librarian. Specify: the three layers and folder map; file-naming rules; the frontmatter fields every page must carry; the ingest / query / lint workflows step by step; and the privacy rules. Keep it editable — it co-evolves with the store.

Fix the conventions

Decide your categories, your frontmatter schema (e.g. type · category · date · related · tags), and adopt [[wiki-links]] for cross-references. Consistency here is what later makes the store queryable.

Define the workflows as named recipes

Give each workflow a trigger and a fixed sequence of steps (see §06). Naming them — "ingest", "query", "lint" — lets you invoke them with one word and get repeatable behaviour.

Seed and grow agent memory

Write a few starting facts — your preferences, your standing instructions — as one-fact files plus an index. Then let it accrete: whenever you correct the agent, capture the lesson as a feedback fact so it sticks.

Set the privacy boundary

Declare the store local-only by default, and require explicit confirmation before anything leaves the machine. Make this a rule in the schema, not an afterthought.

Run it, then lint it

Drop documents in inbox/ and ingest. After a batch, run a lint pass to catch contradictions and gaps. Let the schema evolve as you discover what conventions you actually need.

The schema is the whole product. Everything distinctive about the system lives in that one instruction file. Tools and models are interchangeable; a clear, opinionated schema is what makes the output consistent, queryable, and trustworthy over months of use.

08Build it with an agent

The checklist above is the manual path. The faster, friendlier one: hand a single playbook to your assistant and let it build the system with you — where building is the design, the onboarding, and the tutorial, all at once.

Give your agent build-with-agent.md and it runs a guided, eleven-phase co-build: it explains each concept, interviews you to tailor the shape, builds the artifact only after you approve a diff, proves it works, then logs what changed and why. You don't end with a folder you don't understand — you end with one you helped make.

Every phase runs the same loop

Predictable rhythm, so you always know where you are and never get a silent change:

TEACH→ TAILOR→ BUILD→ VERIFY→ CHECKPOINT→ LOG

Explain the concept and why it matters · ask only what's needed to tailor it · build it after a confirmed diff · look at the result together · recap and check understanding · record the decision. Stop any time and the system is left working at the last completed phase.

Self-documenting by construction

The build narrates itself as it goes, so the system stays legible long after the session. Two logs, kept distinct:

CHANGELOG.log

The append-only story of building and evolving the system — each entry pairs a decision with the artifact it produced and the why. Written at the LOG beat of every phase.

kb/log.md

The routine operations log — one line per ingest, query, or lint, as the store is used day to day.

Plus the artifacts that explain themselves: SCHEMA.md (anyone who reads it can run the system), frontmatter on every page, and a generated README.md summarising what was built and decided.

Generic until you personalise it. The playbook ships anonymized and tool-agnostic; everything specific — your categories, entities, and examples — comes from you, at build time. Nothing leaves your machine without an explicit yes.

09Extending the system

The core is just files and workflows, so new capabilities attach at the edges without disturbing it. Four are especially high-leverage — two move documents in and knowledge out, two upgrade how you connect and consume the store.

@
Extension · inbound

Email ingestion & attachment processing

Turn a mailbox into an automatic source feed, so documents file themselves.

  1. Connect a mail source via a connector (an email tool/plugin, or an IMAP poll). Watch one dedicated label or folder, not the whole inbox.
  2. For each new message: capture sender, subject, date and body; save every attachment into inbox/.
  3. Run the standard ingest workflow on each attachment. Write the message body itself as a source page too, when it carries information.
  4. Process by type: PDFs read natively, images via OCR, spreadsheets/exports parsed into tables.
  5. Mark the message processed (label/archive) and record its ID, so re-runs never double-ingest.
⌕
Extension · outbound

Deep-research integration

Let the agent enrich the store with external context — verified and cited.

  1. Add a research workflow: take a question, fan out parallel web searches, fetch and read the best sources.
  2. Adversarially verify key claims across independent sources before accepting them.
  3. Synthesise a cited report and save it to queries/ with frontmatter, linked to the relevant entities and topics.
  4. Use it to answer "what are my options for X" questions and fold the conclusion back into the knowledge base.
G
Extension · the concrete engine

Google Workspace via gog

The generic "mail source" above, made real. gog (a.k.a. gogcli) is a fast, JSON-first CLI for Gmail, Drive, Calendar, Docs and Sheets — OAuth sign-in, multi-account, and shipped as an agent skill, so it scripts straight into your workflows.

  1. Gmail → poll one label; pull message bodies and save attachments into inbox/, then run ingest.
  2. Drive → mirror a folder as a source feed — new files in Drive become new sources automatically.
  3. Calendar ← push a document's deadlines and required actions out as events; or pull events in as dated context.
  4. Docs / Sheets → read structured exports directly, no manual download step.
  5. JSON output means every result is parseable — record processed IDs to stay idempotent.

→ ready-made build prompt for the Gmail flow in §10 · /ingest-email.

◆
Extension · the human layer

Browse it as an Obsidian vault

The store was designed for this: open kb/ as a vault and the conventions light up. The agent writes the files; you read, navigate, and query them.

  1. Graph view renders the [[wiki-links]] as a navigable map of entities, topics and sources.
  2. Dataview / Bases run live queries over the YAML frontmatter — "every source in category X since date Y" — exactly what the frontmatter spec was built for.
  3. Backlinks surface every page that references the one you're reading.
  4. Templates & Canvas mirror the source-page template and let you assemble visual boards; mobile apps + your own sync keep it with you.
  5. Zero lock-in — it's plain Markdown, readable with or without Obsidian.
Mind the trust boundary. The inbound and outbound extensions point in opposite directions. Ingestion — whether generic or via gog — only pulls in; it should never reply, forward, or send. Deep research reaches out to the open web, so research questions must be phrased generically and must never carry your private source content into a search query. Obsidian is purely local. Keep these flows separate by design, and scope any cloud connector to read-only where you can.

Further extensions

  • Specialised sub-agents — task-focused agents that read only a defined slice of the store (e.g. a filing assistant, or a per-topic analyst), so each works from exactly the right context.
  • Scheduled runs — a periodic job that ingests whatever has arrived and runs a lint pass, so the store maintains itself.
  • Semantic search — an embeddings index over the synthesis layer for fuzzy "where did I see…" retrieval, complementing the exact frontmatter queries.
  • OCR & transcription — pipelines that turn scans, photos, and audio into text before ingest, widening what counts as a "source".

10Guided setup — the onboarding interview

The fastest way to stand up a new instance is to let the agent do it: interview the user, then generate a scaffold tailored to their answers. This is itself an agent instruction you can drop into a /setup skill.

A blank folder and a generic schema are a cold start. Instead, the agent runs a short structured interview — one question at a time, reflecting answers back — to learn the shape of the store, its area of focus, the goal it serves, which extensions to wire, and which skills to create. From the answers it produces the folder scaffold, a SCHEMA.md written for that domain, a few seed memory facts, and a recommended tools plan.

drop-in instruction · paste into your schema or a /setup skill

ROLE. You are setting up a new two-memory knowledge base for a first-time
user. Run the interview below one topic at a time; reflect each answer back
before moving on. Never invent categories or entities the user didn't confirm.

INTERVIEW
  1 · SHAPE & SCOPE  What body of documents should this cover, and roughly
                    how many to start? Just you, or shared with others?
  2 · AREA OF FOCUS  What domains live inside it? Name 4–8 categories for
                    raw/ — we can always add more later.
  3 · GOAL          What decisions or questions should this help with?
                    What does "this is working" look like in three months?
  4 · ENTITIES      Who or what recurs — people, organisations, accounts —
                    that deserves its own page?
  5 · EXTENSIONS    Which to wire now?
                    [ ] Obsidian vault     [ ] gog (Gmail / Drive / Calendar)
                    [ ] deep research      [ ] scheduled ingest + lint
  6 · SKILLS        Which one-word workflows? Defaults: /ingest /query /lint.
                    Add /digest (periodic summary)? /setup (re-run this)?
  7 · PRIVACY       Confirm: local-only, with explicit confirmation before
                    anything leaves the machine — or state your exceptions.

THEN PRODUCE (show a diff before writing anything)
  • scaffold:  inbox/  raw/<category>/  kb/{entities,topics,sources,queries}
  • SCHEMA.md tailored to the answers — layers, naming, the frontmatter spec,
    the ingest / query / lint recipes, and the privacy rule
  • 2–4 seed agent-memory facts capturing stated preferences (user / feedback)
  • INDEX.md + a stub overview.md
  • a recommended skills & tools plan (below), flagged exists vs build —
    then offer to scaffold the "build" ones

RULES. One question at a time · confirm before filing · keep SCHEMA.md
editable · prefer the user's words for category and entity names.

Skills to build — with a prompt for each

None of these ship ready-made; each is a small skill the agent scaffolds for you. Hand it the build prompt and it creates the skill, wired to your SCHEMA.md.

/ingest-emailskill

Drains a Gmail label hands-off: pulls tagged messages, files the body, dumps attachments into the inbox, then marks each done.

build promptCreate an ingest-email skill that uses the gog CLI to drain a Gmail
label end to end:
1. List messages labelled ingest-pending
   (e.g. gog gmail messages list --label ingest-pending --json).
2. For each message, oldest first:
   a. Write its body into the knowledge base as a kb/sources/ page —
      subject, sender, date, body — following the ingest workflow.
   b. Download every attachment into inbox/
      (gog gmail attachments download <id> --out inbox/).
   c. Run /ingest over the new inbox/ files so each attachment is
      classified, filed into raw/, and summarised.
3. On success, relabel: remove ingest-pending, add ingest-done
   (gog gmail messages modify <id> --remove-label ingest-pending --add-label ingest-done).
Rules: pull-only — never delete, reply, or forward. Record processed
message IDs for idempotency; skip anything already ingest-done; if a
message fails, leave its label untouched so the next run retries it.
(gog flags are indicative — confirm against your installed gog version.)

11Automating the compile loop

Everything so far treats writing to agent memory as a manual act — you notice a durable lesson and save it. It can be automated end to end.

build on
The cleanest published take on this is coleam00/claude-memory-compiler, which turns assistant conversations into a compiled knowledge base. Its moves transplant directly onto the agent-memory side of this architecture — credit to that project for the pattern.

Auto-capture with hooks

Session-boundary hooks (at session end, and before context is compacted) read the transcript and append raw observations to a daily log — daily/YYYY-MM-DD.md. Capture becomes automatic; nothing depends on remembering to save.

Decouple capture from compilation

Capture is cheap and continuous; compilation is batched and unattended. A scheduled compile.py (say, nightly) distils the day's logs into structured, deduplicated concept articles — the same read/write compile of §04, run on a timer instead of inline.

Retrieve through the index

A query.py answers questions by reading the catalog, not a vector store — fast, legible, and debuggable.

Lint with many validators

A lint.py runs a battery of checks — broken links, orphaned articles, contradictions, staleness — the automated form of the Lint workflow in §06.

Design rule — index before embeddings. At personal scale (roughly 50–500 notes, and comfortably up to ~2000), an agent reading a structured index.md outperforms vector similarity search: simpler, cheaper, and far more legible. Only reach for embeddings or a RAG store once the index itself no longer fits comfortably in context. This is exactly why the whole system is plain files, not a database.

An optional refinement from the same project: split the synthesis into concepts/ (the articles), connections/ (explicit cross-references), and qa/ (captured question–answer pairs) — a finer-grained shape for the compiled layer once it grows.

12Design principles

The handful of rules that keep the system trustworthy as it scales.

  • Two memories, kept apart. Method in agent memory, matter in the knowledge base. Don't let one bleed into the other.
  • Sources are immutable. The synthesis can be rewritten freely; the originals never change. This is what lets you trust the store.
  • Everything is plain files. Markdown, frontmatter, folders, version control. No database to migrate, nothing you can't open by hand.
  • Observe before interpreting. When data is ambiguous, record what's visible and ask — don't let a plausible guess harden into a recorded "fact".
  • Local-first, explicit egress. Nothing leaves the machine without a deliberate, confirmed step. Outward-facing features are the exception, fenced off and clearly labelled.
  • The schema co-evolves. Treat the constitution as living; refine it whenever you discover a convention you wish you'd had.
